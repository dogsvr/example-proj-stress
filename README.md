# example-proj-stress

Stress-testing harness for [`example-proj`](../example-proj/). Treats the load
generator as a **headless stress client** — parallel to `example-proj-client`
(the Phaser browser client) — and reuses the same `example-proj/protocols/*`
subpath via a `file:` dep, so server-side protocol changes propagate without a
publish step.

Tested on Node.js v24.13.0 on Linux (x86-64); other maintained LTS lines
expected to work but not routinely exercised.

## Layout

Four pieces sit side-by-side under this repo, each independently usable:

- `bots/` — Node.js bot client (`stress-bots` npm package). Talks tsrpc + Colyseus
  to dir/zonesvr/battlesvr, exposes its own `/metrics` on `:9201`. Four
  scenarios: A room capacity, B login/battle, C worker scaling, D hot update.
- `ghz/` — gRPC stress configs targeting battlesvr / zonesvr inner endpoints.
- `k6/` — HTTP-side load (`dir/Common`-routed tsrpc-over-HTTP).
- `observability/` — `grafana/otel-lgtm` single-container backend (Loki +
  Grafana + Tempo + Prometheus + OTel Collector). Holds `prometheus.yml`,
  Grafana dashboards, and the dashboard provider yml — all mounted into the
  container. Scrapes dir/zonesvr/battlesvr main + worker `/metrics` plus the
  bots' `:9201`, and receives OTLP traces/logs on `:4317`/`:4318`.

`RUNBOOK.md` (top-level) is the end-to-end manual recipe — start servers, fill
seed data, run scenarios, collect reports. Each subdirectory has its own
focused README.

## Build the bots

```sh
cd bots
npm install              # resolves "example-proj": "file:../../example-proj"
npm run build            # emits dist/scenarios/*.js
```

The `example-proj` `file:` dep does **not** trigger an example-proj rebuild —
its `dist/` must already exist (run `cd ../example-proj && npm run build`
first). The bots only consume `example-proj/protocols/*` (TS-generated by
`cl-tsrpc protoc`), so the protocol files need to be present in
`example-proj/dist/protocols/`.

## Seed data lives in example-proj

Bots match the `openId = stress_<seq>` naming scheme produced by
`example-proj`'s `npm run ops -- stress:fill`. The seeder is intentionally
left in the server repo because it understands `role_coll` schema, the rank
ZSET encoding (via `@dogsvr/cfg-luban`), and the gid range allocation — all
server-side concerns. The cross-repo contract is just the openId naming.

```sh
cd ../example-proj
npm run ops -- stress:fill --count 10000 --zone-id 1 --yes
# ... run scenarios from this repo ...
npm run ops -- stress:unfill --count 10000 --yes
```
